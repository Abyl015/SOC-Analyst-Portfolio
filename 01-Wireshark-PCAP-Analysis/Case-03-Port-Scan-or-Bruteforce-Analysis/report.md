# Case 03 - Port Scan Reconnaissance Analysis

## Overview

This report documents a local lab investigation of port scanning and reconnaissance activity.

A Windows-based scanner host performed Nmap scans against a Windows target machine. Wireshark was running on the target host and captured the network traffic generated by the scan.

## Lab Environment

| Role | IP Address | Description |
|------|------------|-------------|
| Scanner / Attacker | `192.168.1.57` | Windows host running Nmap |
| Target / Victim | `192.168.1.49` | Windows host monitored with Wireshark |

## Nmap Commands Used

```cmd
nmap -sT 192.168.1.49 -oN case03-basic-scan.txt
nmap -sV 192.168.1.49 -oN case03-service-scan.txt
nmap -A 192.168.1.49 -oN case03-aggressive-scan.txt
```

## Executive Summary

The scan activity originated from `192.168.1.57` and targeted `192.168.1.49`.

Nmap identified the target host as active and found multiple open TCP ports. Wireshark confirmed the scanning behavior through repeated TCP SYN packets from the scanner to multiple destination ports on the target.

## Target Details

| Field | Value |
|-------|-------|
| Target IP | `192.168.1.49` |
| Target MAC Address | `98:FA:9B:00:0A:27` |
| Hostname | `IT-01` |

## Scanner Identification

The scanner IP address was identified by filtering incoming TCP SYN packets to the target host.

```wireshark
ip.dst == 192.168.1.49 and tcp.flags.syn == 1 and tcp.flags.ack == 0
```

The filtered traffic showed repeated SYN packets from `192.168.1.57` to `192.168.1.49` across multiple destination ports. This pattern is consistent with Nmap port scanning activity.

## Open Ports Identified

| Port | State | Service | Details |
|------|-------|---------|---------|
| `80/tcp` | Open | HTTP | Apache httpd 2.4.66 on Windows |
| `135/tcp` | Open | MSRPC | Microsoft Windows RPC |
| `139/tcp` | Open | NetBIOS-SSN | Microsoft Windows NetBIOS session service |
| `445/tcp` | Open | SMB | Microsoft Windows microsoft-ds |
| `515/tcp` | Open | Printer | Microsoft LPD |
| `3389/tcp` | Open | RDP | Microsoft Terminal Services |
| `5357/tcp` | Open | HTTPAPI | Microsoft HTTPAPI 2.0 |
| `7070/tcp` | Open | SSL / Remote Access | AnyDesk Client certificate observed |

## Wireshark Evidence

### SYN Packets

```wireshark
ip.dst == 192.168.1.49 and tcp.flags.syn == 1 and tcp.flags.ack == 0
```

This showed that `192.168.1.57` repeatedly initiated TCP connection attempts to multiple ports on `192.168.1.49`.

### SYN/ACK Responses

```wireshark
ip.src == 192.168.1.49 and ip.dst == 192.168.1.57 and tcp.flags.syn == 1 and tcp.flags.ack == 1
```

SYN/ACK responses from the target indicated open TCP ports.

### Reset Responses

```wireshark
ip.src == 192.168.1.49 and ip.dst == 192.168.1.57 and tcp.flags.reset == 1
```

RST and RST/ACK packets indicated that some connection attempts were refused or reset by the target host.

## Notable Security Findings

### Finding 1 - RDP Exposed

Port `3389/tcp` was open and identified as Microsoft Terminal Services.

Risk:

- brute-force attacks;
- unauthorized remote access attempts;
- increased exposure to ransomware-related attack chains.

Recommendation:

- restrict RDP using firewall rules;
- require VPN for remote access;
- enforce strong authentication;
- monitor failed RDP logon attempts.

### Finding 2 - SMB Exposed

Port `445/tcp` was open and identified as Microsoft SMB.

Risk:

- lateral movement;
- unauthorized file share access;
- credential relay attacks;
- internal reconnaissance.

Recommendation:

- restrict SMB access to trusted systems;
- require SMB signing where possible;
- monitor SMB authentication events.

### Finding 3 - Apache HTTP Server Exposed

Port `80/tcp` was open and identified as Apache httpd `2.4.66` on Windows. The aggressive scan also reported the HTTP `TRACE` method as potentially risky.

Recommendation:

- disable HTTP TRACE if not required;
- review Apache configuration;
- restrict access if the web service is not intended for internal use.

### Finding 4 - AnyDesk-Related Service Observed

Port `7070/tcp` showed an SSL certificate with common name `AnyDesk Client`.

Recommendation:

- verify business need for AnyDesk;
- restrict remote access connections;
- monitor remote access logs;
- remove or disable if not required.

## Risk Analysis

The target host exposes several services that can increase the attack surface of the system. While the activity in this case was generated in a controlled lab environment, the same traffic pattern in a production network would indicate reconnaissance and possible pre-attack activity.

## Severity Assessment

| Category | Assessment |
|----------|------------|
| Severity | Medium |
| Confidence | High |
| Scanner IP | `192.168.1.57` |
| Target IP | `192.168.1.49` |
| Main Activity | TCP port scanning / reconnaissance |
| Required Action | Review exposed services and monitor scanner activity |

## Recommended SOC Actions

- Confirm whether the scan from `192.168.1.57` was authorized.
- Investigate the scanner host if the scan was not expected.
- Review firewall and endpoint logs on `192.168.1.49`.
- Monitor repeated SYN packets to multiple destination ports.
- Create SIEM detection rules for internal port scanning.
- Restrict RDP and SMB access.
- Disable unnecessary services.
- Review AnyDesk usage and exposure.
- Harden HTTP and SMB configurations.

## Conclusion

The analysis confirmed port scanning activity from `192.168.1.57` against `192.168.1.49`.

Wireshark showed repeated TCP SYN packets from the scanner to the target across multiple destination ports. SYN/ACK responses from the target confirmed open ports, while reset packets indicated closed or refused connections.

This case demonstrates practical SOC Analyst skills in identifying reconnaissance activity, analyzing PCAP traffic, interpreting Nmap scan results, and preparing defensive recommendations.
